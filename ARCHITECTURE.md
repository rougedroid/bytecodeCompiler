# Architecture and Design Overview

This document provides a high-level view of the ByteCode Compiler's architecture, design decisions, and how all components interact.

## System Architecture

```
┌─────────────────────────────────────────────────────────────┐
│                      ByteCode Compiler                       │
├─────────────────────────────────────────────────────────────┤
│                                                               │
│  ┌──────────────────────────────────────────────────────┐   │
│  │            SOURCE PROGRAM (Imperative Language)       │   │
│  │  Variables, Arithmetic, Conditionals, Loops, Return  │   │
│  └──────────────────────────────────────────────────────┘   │
│                           ↓                                   │
│  ┌──────────────────────────────────────────────────────┐   │
│  │          FRONT END: Tokenization & Classification    │   │
│  │  • load_code()      — Load source program            │   │
│  │  • tokenizer()      — Split into tokens              │   │
│  │  • classifier()     — Assign token types             │   │
│  └──────────────────────────────────────────────────────┘   │
│                           ↓                                   │
│  ┌──────────────────────────────────────────────────────┐   │
│  │        PARSER: Syntax Analysis & AST Construction    │   │
│  │  • parse_program()  — Top-level program entry        │   │
│  │  • parse_statement()— Variable, if, while, return    │   │
│  │  • parse_expression()— Binary arithmetic ops         │   │
│  │  • parse_logic()    — Comparisons for control flow   │   │
│  └──────────────────────────────────────────────────────┘   │
│                           ↓                                   │
│  ┌──────────────────────────────────────────────────────┐   │
│  │     CODE GENERATION: AST to Bytecode Translation     │   │
│  │  • translate()      — Convert expressions to opcodes │   │
│  │  • Bytecode emission— Write 16-bit instructions      │   │
│  └──────────────────────────────────────────────────────┘   │
│                           ↓                                   │
│  ┌──────────────────────────────────────────────────────┐   │
│  │           OUTPUT: Generated Bytecode Stream           │   │
│  │  Instruction sequence ready for VM execution         │   │
│  └──────────────────────────────────────────────────────┘   │
│                                                               │
└─────────────────────────────────────────────────────────────┘
```

## Data Flow

### 1. Input Processing

```
Source Code String
    ↓
tokenizer()  ──→ Token Array (space-separated tokens)
    ↓
classifier() ──→ Token Types (keywords, values, operators, etc.)
```

### 2. Parsing

```
Token Stream
    ↓
parse_program() ──→ Statement Sequence
    ↓
parse_statement()
    ├──→ Variable Assignment ──→ AST Expression Node
    ├──→ If/Else Statement ──→ Control Flow AST
    ├──→ While Loop ──→ Loop Control AST
    └──→ Return Statement ──→ Return Node
    ↓
Variable Pool Management (track declared variables & registers)
```

### 3. Code Generation

```
AST Nodes
    ↓
translate() ──→ Bytecode Instructions
    ↓
ByteStream ──→ 16-bit Word Sequences
    ↓
Output (printed to stdout)
```

## Core Components

### Component 1: Token Management

**Purpose:** Break source text into meaningful units

**Key Functions:**
- `tokenizer()` — Split by whitespace
- `classifier()` — Assign semantic types

**Key Types:**
- `token_t` — Single token with type info
- `token_array_t` — Dynamic array of tokens

**Allocation:** Dynamically resized array

### Component 2: Variable Management

**Purpose:** Track variable names and virtual register assignments

**Key Functions:**
- `lookup_var()` — Find variable by name
- `add_var()` — Create new variable with register

**Key Types:**
- `variable_t` — Name + register pair
- `var_pool_t` — Dynamic pool of variables

**Allocation:** Dynamically resized array

### Component 3: Expression Parser

**Purpose:** Parse arithmetic and comparison expressions

**Key Functions:**
- `parse_primary()` — Values, variables, parenthesized expressions
- `parse_term()` — Multiplication and division (*)
- `parse_expression()` — Addition and subtraction (+, -)
- `parse_logic()` — Comparisons (<, >, ==)

**Output:** AST node representing expression

**Limitations:** Operator precedence not fully standard

### Component 4: Statement Parser

**Purpose:** Parse complete statements (assignments, control flow, return)

**Key Functions:**
- `parse_statement()` — Dispatch to appropriate statement type
- `parse_program()` — Loop over all statements until EOF

**Statements Supported:**
- Variable assignments
- If/else conditionals
- While loops
- Return statements

### Component 5: Code Generator

**Purpose:** Convert AST to bytecode instructions

**Key Functions:**
- `translate()` — Recursively convert expression nodes
- Bytecode emission — Write instructions to stream

**Key Types:**
- `ASTNode_t` — Expression tree node
- `ByteStream_t` — Dynamic array of 16-bit instructions

**Output:** Bytecode stream ready for execution

## Design Decisions

### 1. Single-Pass Compilation

**Decision:** Parse and generate bytecode in a single pass

**Rationale:**
- Simpler implementation for educational purposes
- No need for separate AST storage phase
- Memory efficient for small programs

**Trade-offs:**
- Cannot perform multi-pass optimizations
- Error recovery is difficult
- Limited code generation flexibility

### 2. Recursive-Descent Parsing

**Decision:** Use recursive function calls for parsing

**Rationale:**
- Natural mapping between grammar and functions
- No parser generator required
- Easy to understand and debug

**Trade-offs:**
- Not suitable for ambiguous grammars
- Left recursion must be avoided
- Cannot handle operator precedence perfectly

### 3. Dynamic Memory Allocation with Doubling

**Decision:** Use `malloc()` and `realloc()` with geometric growth

**Rationale:**
- Standard C approach
- Provides O(1) amortized append complexity
- Reduces reallocation frequency

**Trade-offs:**
- Wastes memory (unused capacity)
- Memory leaks if not explicitly freed
- Fragmentation over long runs

### 4. Virtual Register Model

**Decision:** Assign each variable a virtual register address

**Rationale:**
- Decouples source variables from target registers
- Simplifies code generation
- Natural fit for bytecode architecture

**Trade-offs:**
- Requires register allocation phase
- Current implementation is naive (sequential assignment)
- No optimization or spilling logic

## Control Flow Implementation

### If/Else Statements

```
    parse_logic()  ──→ Compare condition
         ↓
    if true:  emit JUMP to else_label
    generate_if_body()
    emit JUMP to end_label
    emit else_label:
    generate_else_body()
    emit end_label:
```

### While Loops

```
    emit loop_start_label:
    parse_logic()  ──→ Compare condition
    if false:  emit JUMP to loop_end_label
    generate_loop_body()
    emit JUMP back to loop_start_label
    emit loop_end_label:
```

## Memory Layout

### Compiler-Side Memory

```
┌─────────────────────────────────┐
│  Token Array                    │  Dynamic allocation
├─────────────────────────────────┤
│  Variable Pool                  │  Dynamic allocation
├─────────────────────────────────┤
│  Bytecode Stream                │  Dynamic allocation
├─────────────────────────────────┤
│  Temporary AST Nodes            │  Stack allocation
├─────────────────────────────────┤
│  Global State & Parsing Context │  Stack allocation
└─────────────────────────────────┘
```

```
┌─ 0x0000 ──────────────────────┐

├─ 0xFFF3 ──────────────────────┤

```

## Supported Language Features

### Fully Supported

- ✅ Variable declarations (implicit on first assignment)
- ✅ Arithmetic: +, -, *, /
- ✅ Comparisons: <, >, ==
- ✅ Control flow: if/else
- ✅ Control flow: while loops
- ✅ Return statements
- ✅ Parenthesized expressions
- ✅ Integer literals (16-bit signed range)

### Not Supported

- ❌ Function definitions or calls
- ❌ Arrays or data structures
- ❌ String literals or text operations
- ❌ Logical operators (&&, ||, !)
- ❌ Bitwise operations
- ❌ Type declarations or casting
- ❌ Scoping (all variables global)
- ❌ Comments
- ❌ Break/continue in loops

## Performance Characteristics

### Compilation Time
- Linear in program size: O(n) where n = token count
- No optimization passes

### Memory Usage
- Token array: grows with input size
- Variable pool: grows with unique variables
- Bytecode stream: grows with program complexity

### Bytecode Size
- Each instruction is 2-4 bytes (16-bit words)
- No compression or optimization
- Typical program: kilobytes

## Extension Points

### Easy to Add Features

1. **New Operators:** Add cases to classifier, parse_expression, translate
2. **New Keywords:** Add to keyword list, parser cases
3. **Optimization:** Add post-processing to bytecode
4. **New Data Types:** Extend variable_t structure

### Harder to Add Features

1. **Functions:** Requires call stack, scope management
2. **Arrays:** Requires heap allocation, index calculations
3. **Proper Precedence:** Parser restructuring needed
4. **Error Recovery:** Multiple passes needed

## Summary

The ByteCode Compiler demonstrates a complete compilation pipeline:
- **Tokenization** breaks source into units
- **Parsing** builds semantic representation
- **Code generation** produces target instructions
- **Memory management** uses dynamic allocation safely
- **Modular design** allows incremental improvements

The architecture prioritizes clarity and simplicity over performance, making it ideal for educational and prototyping purposes.
