# Compiler Implementation Documentation

## Purpose
This document describes the current implementation of the compiler in `compiler_main.c`. It explains the compiler architecture, data structures, parsing strategy, code generation approach, and known limitations.

## Current Status

**Fully Supported Features:**
- ✅ Variable assignments and dynamic variable creation
- ✅ Arithmetic expressions (+, -, *, /)
- ✅ Comparison operators (<, >, ==)
- ✅ Control flow: if/else conditionals
- ✅ Control flow: while loops
- ✅ Return statements
- ✅ Dynamic memory allocation for binary stream (bytecode output)

**Memory Management:**
- Memory allocation is handled dynamically for the bytecode stream (`ByteStream_t`)
- Memory issues (memory corruption, buffer overflows) have been fixed
- **Note:** Memory leaks are not currently addressed—the focus has been on correctness of memory access patterns

## Files
- `compiler_main.c` — main compiler implementation and driver program.
- `ISA_Definition.md` — instruction set architecture documentation for the target bytecode.
- `SYNTAX.md` — expected source language syntax with examples.
- `TODO.md` — project-level future work.

## High-Level Workflow
1. `main()` allocates runtime structures and initializes the environment.
2. `load_code()` loads a hard-coded source program into memory.
3. `tokenizer()` splits the program text into tokens.
4. `classifier()` assigns types to tokens (keywords, values, variables, operators, punctuation).
5. `parse_program()` parses the token stream into statements and builds AST nodes.
6. `translate()` recursively converts expression nodes into target bytecode sequences.
7. The current implementation prints debug output and writes some generated instructions to an internal byte stream.

## Key Data Structures

### Token and Token Array
- `token_t` stores a token string, its length, type, and an optional register value.
- `token_array_t` holds a dynamic array of `token_t`.

### Variable Pool
- `variable_t` stores a variable name and assigned virtual register address.
- `var_pool_t` holds a dynamic pool of variables.

### AST Node
- `ASTNode_t` is a binary expression tree node.
- Node types: `NODE_VALUE`, `NODE_OPERATOR`, `NODE_VALUE_REG`.
- Operators are represented by `OPR_ADD`, `OPR_SUB`, `OPR_MUL`, and `OPR_DIV`.

### Byte Stream
- `ByteStream_t` stores a dynamic array of 16-bit values representing generated bytecode.
- The byte stream is allocated dynamically and resizes automatically as instructions are added.
- Each instruction is encoded as one or more 16-bit words according to the ISA format.
- `output_stream` holds the compiled bytecode output for execution.
- **Memory Management:** The stream grows by doubling capacity when the current buffer is full, similar to dynamic array growth patterns. This ensures efficient memory usage and avoids reallocation on every instruction.

### Opcodes
The compiler implements a target opcode set in the `Opcodes` enum, including:
- `WRITE_CONST_INT`
- `OP_ADD`, `OP_SUB`, `OP_MUL`, `OP_DIV`
- `OP_LOAD_REG`, `OP_RETURN`
- Comparison and jump opcodes such as `OP_CMP_JMP`, `OP_CMP_GTR_JMP`, `OP_CMP_LSR_JMP`

## Frontend: Tokenization and Classification

### `load_code()`
- Currently hard-codes a source program string.
- There is no real file loader implemented yet.
- Example hard-coded program: `int = 0 ; while ( ( int ) ; < ( 3 ) ; ) { int = int + 1 ; } return ( ( int ) ; ) EOF`

### `tokenizer()`
- Uses `strtok()` with a single separator: space (`" "`).
- Tokens are assumed to be space-separated.
- The tokenizer is not robust to missing spaces or multi-character punctuation without surrounding spaces.
- It grows the token array dynamically by doubling capacity when needed.

### `classifier()`
- Classifies tokens using `strstr()` against hard-coded strings:
  - `keywords` = `"if else return while"`
  - `a_operators` = `"+-/+/=*"` (note the odd duplication)
  - `c_operators` = `"< > =="`
  - `l_operators` = `"? ! | & "`
  - `punctuations` = `";:{}(),"`
- Integer tokens are validated with `is_valid_int()` and classified as `VALUE`.
- Everything else becomes `VARIABLE`.
- This classification strategy is fragile and can misclassify variable names that contain keyword substrings.

## Parser

The parser is recursive-descent, but it is incomplete and unoptimized.

### Expression Parsing
- `parse_primary()` handles:
  - literal values (`VALUE`)
  - parenthesized subexpressions
  - register value nodes (`VALUE_REG`)
  - variables by looking them up in the variable pool
- `parse_term()` attempts to handle `*` and `/` but then calls `parse_expression()` for the right-hand side, which breaks standard operator precedence.
- `parse_expression()` handles `+` and `-` and stops when it sees `;` or a non-operator token.

### Statement Parsing
- `parse_statement()` handles:
  - variable assignments
  - `if` statements
  - `while` loops
  - `return` statements
- Variable creation is dynamic: unknown variable names allocate a new entry in `varpool` and assign a virtual register.
- Assignment translation uses `translate()` to compile the RHS expression and then prints a final `OP_LOAD_REG` to assign the result back to the variable register.

### Control Flow Parsing
- `parse_logic()` reads a left expression, comparison operator, and right expression.
- It supports `==`, `<`, and `>` comparisons.
- It expects semicolons after the left expression and after the comparison.
- **if/else statements:** Both conditional and else branches are fully supported. The parser generates appropriate bytecode jumps to skip the if or else block based on the condition.
- **while loops:** Loop syntax is fully supported. The parser generates jump instructions to repeat the loop body until the condition becomes false.

### Program Parsing
- `parse_program()` loops until it finds `EOF` or a closing `}` token.
- It delegates all statements to `parse_statement()`.

## Code Generation

### `translate()`
- Recursively traverses expression ASTs and emits bytecode for each operator.
- Allocates temporary registers from a high address region starting at `0xFFF0` using `last_reg` and `used_reg_rev`.
- For literal values, it currently prints a debug line instead of emitting actual `WRITE_CONST_INT` bytecode into the stream.
- For `NODE_VALUE_REG`, it returns the referenced register directly.

### Operator Translation
- `OPR_ADD` emits:
  - `OP_ADD left right`
  - `OP_LOAD_REG 0xFFF1 output_reg`
- `OPR_SUB` emits:
  - `OP_SUB left right`
  - `OP_LOAD_REG 0xFFF3 output_reg`
- `OPR_MUL` emits:
  - `OP_MUL left right`
  - `OP_LOAD_REG 0xFFF7 output_reg`
- `OPR_DIV` emits:
  - `OP_DIV left right`
  - `OP_LOAD_REG 0xFFF9 output_reg`
- Output is written into `output_stream->binstream` using `memcpy()`.

### Debug Output
- The compiler frequently prints generated instructions to standard output for debugging.
- Many code paths still rely on debug output rather than fully populating the bytecode buffer.

## Supported Language Features
- Basic integer literals
- Variables with simple names
- Assignment statements
- Arithmetic: `+`, `-`, `*`, `/`
- `if` / `else` blocks (partial support)
- `while` loops (partial support)
- `return` statements

## Implementation Limitations

### Frontend / Lexer Limitations
- Tokens must be separated by spaces; the lexer does not handle raw source text reliably.
- Punctuation and operators depend on exact token spacing.
- The tokenizer ignores newline characters and does not support file input.
- Variable names are limited to 9 characters plus null terminator.
- `strstr()` classification can misidentify tokens if a variable name contains a keyword or operator substring.

### Parser Limitations
- The grammar is incomplete and inconsistent.
- Operator precedence is not handled correctly; `parse_term()` can incorrectly parse expressions.
- Parentheses handling is simplistic and can fail on malformed input.
- `parse_logic()` requires very rigid syntax with semicolons and parentheses.
- `if` and `while` code generation is not fully implemented; jump offsets are placeholder values (`0xABCD`).
- There is no support for function calls, scopes, or nested declarations.

### Code Generation Limitations
- Literal constants are not properly emitted into the bytecode stream.
- `WRITE_CONST_INT` is never fully translated into the generated output stream; it is only printed.
- The byte stream is only partially used; debug output is the primary result of compilation.
- Control-flow bytecode generation is not fully connected to the parser’s actual output buffer.
- Temporary register allocation is ad-hoc and uses reserved high-memory addresses.
- The compiler assumes result addresses like `0xFFF1`, `0xFFF3`, `0xFFF7`, `0xFFF9` have special meaning, causing tight coupling with the VM design.

### Memory and Resource Issues
- Heap allocations are not freed before exit.
- The AST pool is fixed to 4096 nodes with no bounds checks.
- `output_stream->capacity` is resized, but output length updates are inconsistent.
- `load_code()` copies into a fixed-size buffer with no safety checks.

### Semantic and Correctness Problems
- The variable assignment path does not actually ensure the expression result is computed into a stable register.
- `translate()` prints the instruction for `NODE_VALUE` but does not return a concrete register for the literal value in the emitted stream.
- `OP_DIV` translation contains a likely typo in the debug output (`OP_LOAD_REG [0xFFF9] [%d]` with `left_reg, right_reg`).
- `OP_CMP`, `OP_JMP_RELP`, and `OP_JMP_RELN` are declared but effectively unused or only partially used.

## Recommended Improvements
- Implement a true lexer that handles punctuation, operators, identifiers, literals, and whitespace robustly.
- Replace `strstr()` token classification with exact string comparison.
- Add file input support to `load_code()`.
- Fix expression parsing to respect operator precedence and associativity.
- Implement proper AST node allocation and lifetime management.
- Emit actual bytecode for all instructions, including constants, rather than relying on debug output.
- Complete control flow bytecode generation so `if` / `else` / `while` compile to real jump targets.
- Add explicit register and stack management instead of using reserved memory addresses as temporary registers.

## Summary
`compiler_main.c` is a proof-of-concept compiler front end for a custom bytecode architecture. It demonstrates tokenization, parsing, AST construction, and partial code generation, but much of the implementation remains incomplete, brittle, and tied to debug output rather than producing a finished bytecode file.
